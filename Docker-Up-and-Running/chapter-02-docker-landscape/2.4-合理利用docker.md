# 2.4 充分理解 Docker

> **后端学习提示**：本章**必学**。建立 Registry、C/S 架构、分层镜像、无状态设计等全局观；是 Ch04 镜像与 Ch08 Compose 的理论基础。

* **核心主旨与实用要点**
  * 澄清开发者初接触 Docker 时的常见误区，深入解析其在无状态设计、轻量级运行和安全隔离边界上的底层哲学。
* **知识点分层与关键结论**
  * **无状态的应用（Stateless Application）**：
    * *核心原则*：Docker 强制要求应用采用无状态设计。这意味着数据库引擎等带状态服务直接放在 Docker 基础架构中属于"逆流而上"，极其容易因为容器的销毁而丢失数据。
    * **状态外存**：强烈建议将应用的状态剥离，存储在外部系统（如数据库、Memcache 实例，或者 S3 等外部块存储设备）中。配置信息则通过**环境变量**在启动时动态传入容器。
  * **容器不是虚拟机（底层逻辑）**：
    * *本质差异*：虚拟机旨在模拟完整硬件和操作系统，需运行数月乃至数年；而 Docker 容器实质是对 **UNIX 进程的轻量级包装**，启动和销毁速度极快，生命周期极短，且往往用完即焚。
  * **极其轻量级的存储占用（案例数据）**：
    * 基于黄金映像（golden image）的虚拟机动辄占用数 GB 空间，而在现有基础映像之上新建的一个 Docker 容器，仅需占用约 **12KB** 的硬盘空间。
  * **实现不可变的基础设施（Immutable Infrastructure）**：
    * *痛点与方案*：传统配置管理需不断维护陈旧的系统状态。使用 Docker 后，当需要升级软件或操作系统时，不再执行系统内升级命令，而是**直接停用旧容器，部署包含新层级的全新容器**。
  * **有限的隔离（安全警告与易错点）**：
    * *资源共享*：不同于虚拟机的绝对硬件隔离，容器与宿主机共享同一个物理内核，共用 CPU 和内存资源，以及底层文件系统层。
    * *严重警告*：默认情况下，很多容器使用 **UID 为 0 (root)** 的用户身份启动。由于容器仅是一层薄的权限隔离，一旦发生安全漏洞导致容器越权，攻击者将直接获得宿主机系统的 root 权限。
* **预留拓展补充空间**
  * > *[此处可补充：面对"有限的隔离"，现代生产环境通常如何配合 Linux 内核特性（如 AppArmor, SELinux, User Namespace）来加固容器的安全边界]*

```bash
# 无状态应用：配置通过环境变量注入
docker run -e DATABASE_URL=postgres://... -e REDIS_URL=redis://... myapp:1.2.3
```
