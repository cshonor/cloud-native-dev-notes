# 第2章 Docker 概览 (The Docker Landscape)

> **后端学习提示**：本章**必学**。建立 Registry、C/S 架构、分层镜像、无状态设计等全局观；是 Ch04 镜像与 Ch08 Compose 的理论基础。

## 简化业务流程

* **核心主旨与实用要点**
  * 深度对比传统应用部署流程与 Docker 工作流程，展示 Docker 是如何通过"职责界限的重新划分"来极大降低跨团队协作沟通成本的。
* **核心知识点与逻辑脉络**
  * **传统部署流程的痛点（因果逻辑）**：
    * *现象*：开发请求资源 → 运维配置好交付 → 开发编写部署脚本 → 双方反复调整脚本当中依赖 → 发现新依赖再次介入。
    * *结果*：高度复杂的系统导致每次部署需耗费数天，浪费精力，降低开发创新能力，导致不得不采用"大版本"发布以减少痛苦。
  * **Docker 改进的工作流程（核心结论）**：
    * Docker 引入了**映像仓库（Registry）**作为中转站，将构建和操作的职责彻底分开。
    * *新流程*：1. 开发者在本地构建 **Docker 映像**，并将其推送到注册处；2. 运维工程师仅负责提供运行容器所需的宿主机资源及基本配置信息；3. 开发者利用上述资源自行完成部署。
  * **功能内置但可更换**：
    * Docker 奉行提供绝大多数人工作所需的内置功能块，同时允许根据具体业务需求轻量级地更换单个功能模块的理念。
* **预留拓展补充空间**
  * > *[此处可补充：在实际的 DevOps 文化落地中，如何利用 Jenkins 或 GitLab CI 自动化衔接这一"开发构建 → 运维提供环境"的标准流水线]*

```
传统流程：开发 ←→ 运维（反复改脚本、调依赖）
Docker 流程：开发 build & push 镜像 → Registry → 运维提供宿主机 → 开发/CI pull & run
```

## 广泛支持和采用

* **核心主旨与实用要点**
  * 说明 Docker 技术的普及程度，其已经成为现代云计算服务提供商和操作系统厂商的通用支持格式。
* **案例数据与核心结论**
  * **云平台支持**：大量大型公有云平台（如 AWS Elastic Beanstalk、Google AppEngine、IBM Cloud、Microsoft Azure 等）均提供对 Docker 的直接支持。
  * **业务价值**：使"**一次编写，到处运行**"的云应用变成现实，帮助大型公司解决业务跨云平台迁移和部署的难题。
  * **操作系统支持**：Docker 客户端可以在绝大多数主流操作系统中运行。尽管 Docker 服务端（守护进程）强依赖于 **Linux** 内核，但几乎各大主流 Linux 发行版（如 Ubuntu）都原生支持 Docker。
* **预留拓展补充空间**
  * > *[此处可补充：当前主流云原生架构下，各大公有云厂商推出的托管 Kubernetes 服务（如 EKS, GKE, ACK）是如何深度集成 Docker/Containerd 运行时的]*

## 架构

* **核心主旨与实用要点**
  * 剖析 Docker 在底层强悍技术之上的极简架构模型，重点拆解其客户端-服务端（C/S）工作方式、网络通信接口及 API 设计。
* **详细知识点拆解**
  * **客户端-服务器模式**：
    * Docker 基础架构由三部分组成：**客户端（Client）**、**服务器（Daemon 守护进程）**以及可选的**注册处（Registry）**。
    * *易错细节*：Docker 不区分客户端和服务器的二进制文件，它们使用的是**同一个**可执行文件（`docker`）。当执行带有 `-d` 参数的命令时，即启动为监听后台的守护进程模式。
  * **网络端口和 UNIX 套接字**：
    * *默认通信机制*：守护进程默认监听一个本地 **UNIX 套接字**进行进程间通信。
    * *TCP 端口规范*：在向 IANA 注册规范后，Docker 默认监听未加密的 **2375** 端口和加密的 **2376** 端口，替代了早期非官方的 4243 端口。
  * **应用程序编程接口（API）**：
    * Docker 守护进程对外暴露了功能丰富的 **RESTful Web API**。
    * *实际应用*：借助 API（如 Go 或 Ruby 语言的封装库），可以轻易将 Docker 的功能集成到自动化脚本、部署工具或调度平台中，使得工具的二次开发极为便捷。
  * **容器的网络层**：
    * *机制原理*：Docker 会创建一个虚拟以太网桥（通常命名为 `docker0`），并将宿主机的端口绑定到容器上。
    * *性能表现*：流经 `docker0` 虚拟网桥的入站流量仅经过一次网络地址转换（NAT），因此虽然是容器，但网络性能极高，吞吐量极大。
* **预留拓展补充空间**
  * > *[此处可补充：Docker 守护进程配置文件 `daemon.json` 的核心参数解析，以及如何通过配置开启 TLS 加密通信保障远程管理安全]*

```
┌─────────┐  UNIX socket / TCP 2375/2376  ┌──────────────┐
│ docker  │ ──────────────────────────────► │ dockerd      │
│ client  │         REST API                │ (Daemon)     │
└─────────┘                               └──────┬───────┘
                                                 │
                    pull / push                  ▼
                                          ┌──────────────┐
                                          │ Registry     │
                                          │ (Docker Hub) │
                                          └──────────────┘
```

| 组件 | 作用 | 后端常用命令 |
|------|------|--------------|
| Client | 发指令 | `docker build/run/push` |
| Daemon | 管理容器/镜像/网络 | 后台服务 |
| Registry | 存储分发镜像 | `docker pull/push` |

## 充分理解 Docker

* **核心主旨与实用要点**
  * 澄清开发者初接触 Docker 时的常见误区，深入解析其在无状态设计、轻量级运行和安全隔离边界上的底层哲学。
* **知识点分层与关键结论**
  * **无状态的应用（Stateless Application）**：
    * *核心原则*：Docker 强制要求应用采用无状态设计。这意味着数据库引擎等带状态服务直接放在 Docker 基础架构中属于"逆流而上"，极其容易因为容器的销毁而丢失数据。
    * **状态外存**：强烈建议将应用的状态剥离，存储在外部系统（如数据库、Memcache 实例，或者 S3 等外部块存储设备）中。配置信息则通过**环境变量**在启动时动态传入容器。
  * **容器不是虚拟机（底层逻辑）**：
    * *本质差异*：虚拟机旨在模拟完整硬件和操作系统，需运行数月乃至数年；而 Docker 容器实质是对 **UNIX 进程的轻量级包装**，启动和销毁速度极快，生命周期极短，且往往用完即焚。
  * **极其轻量级的存储占用（案例数据）**：
    * 基于黄金映像（golden image）的虚拟机动辄占用数 GB 空间，而在现有基础映像之上新建的一个 Docker 容器，仅需占用约 **12KB** 的硬盘空间。
  * **实现不可变的基础设施（Immutable Infrastructure）**：
    * *痛点与方案*：传统配置管理需不断维护陈旧的系统状态。使用 Docker 后，当需要升级软件或操作系统时，不再执行系统内升级命令，而是**直接停用旧容器，部署包含新层级的全新容器**。
  * **有限的隔离（安全警告与易错点）**：
    * *资源共享*：不同于虚拟机的绝对硬件隔离，容器与宿主机共享同一个物理内核，共用 CPU 和内存资源，以及底层文件系统层。
    * *严重警告*：默认情况下，很多容器使用 **UID 为 0 (root)** 的用户身份启动。由于容器仅是一层薄的权限隔离，一旦发生安全漏洞导致容器越权，攻击者将直接获得宿主机系统的 root 权限。
* **预留拓展补充空间**
  * > *[此处可补充：面对"有限的隔离"，现代生产环境通常如何配合 Linux 内核特性（如 AppArmor, SELinux, User Namespace）来加固容器的安全边界]*

```bash
# 无状态应用：配置通过环境变量注入
docker run -e DATABASE_URL=postgres://... -e REDIS_URL=redis://... myapp:1.2.3
```

## Docker 式工作流程

* **核心主旨与实用要点**
  * 梳理 Docker 原生的版本控制逻辑，以及通过 `Dockerfile` 实现"构建-测试-打包-部署"的标准化闭环流水线。
* **工作流拆解与因果关系**
  * **版本控制体系**：
    * **文件系统层（Image Layers）**：Docker 容器由无数个文件系统层堆叠而成，每一层由**唯一的哈希码**标识。
    * *优势因果*：由于采用分层复用架构，在重新构建或拉取映像时，Docker 仅需下载和更新变更的最上层数据。这极大地节省了构建时间、磁盘空间和网络带宽。
    * **映像标签（Tags）**：相当于 Git 中的标签，用于锁定特定版本的构建产物。
    * *易错细节/警告*：在生产自动化部署中，**绝对不要依赖默认的 `latest` 标签**。因为 `latest` 标签会随着每次新版本构建而向前滚动，一旦最新版本存在问题，依赖 `latest` 将导致无法精准回滚到上一个安全的指定版本。
  * **自动化构建（Build）**：
    * 通过编写简单的 `Dockerfile` 脚本并使用 `docker build` 命令，即可自动化生成环境一致的 Docker 映像。`Dockerfile` 的每一行指令通常会生成一个新的映像分层。
  * **测试与打包的革新**：
    * *测试可靠性*：Docker 生成的映像中包含了应用程序及其需要的所有系统级底层依赖。因此在测试环境中通过的映像，可以确保在生产环境中具有绝对一致的表现，排除了底层库版本不一导致的问题。
    * *标准集装箱模式*：构建出的映像就像是标准尺寸的"集装箱"，不论内部运行的是 Node.js、Java 还是 Python 程序，外部的调度工具和平台都只需使用同一种标准方式对其进行搬运和部署。
* **预留拓展补充空间**
  * > *[此处可补充：详细解析 Docker 联合文件系统（UnionFS/Overlay2）是如何在底层实现 Copy-On-Write（写时复制）机制来管理这些映像层的]*

```
Dockerfile → docker build → 镜像(分层) → docker push → Registry
                ↓
         docker run（测试）→ 同一镜像 → 生产部署
```

| 标签策略 | 示例 | 生产环境 |
|----------|------|----------|
| ❌ 避免 | `myapp:latest` | 无法精准回滚 |
| ✅ 推荐 | `myapp:1.2.3` 或 `myapp:sha-abc123` | 可锁定版本 |

## Docker 的生态系统

* **核心主旨与实用要点**
  * 介绍由于单机 Docker 架构的局限性，社区和企业如何围绕其开发衍生工具链，补齐了集群管理、日志和底层操作系统的短板。
* **生态组件拆解**
  * **编排工具（Orchestration Tools）**：
    * *定位*：Docker 自身专注于单机容器管理。为了调度海量跨主机的容器，催生了强大的编排工具市场。
    * *代表产品*：包括 Docker 原生的 **Swarm**、New Relic 开发的 **Centurion**、Google 捐赠的 **Kubernetes** 以及 Apache **Mesos**，这些工具能帮助实现自动化扩展和多节点集群资源分配。
  * **基元宿主机（Atomic Hosts）**：
    * *核心理念*：专为运行 Docker 和 Linux 容器高度精简的底层操作系统发行版。它们专注于容器支持，空间极小。
    * *代表产品*：**CoreOS** 和 **Project Atomic**。它们支持原子的系统更新和回滚，把容器的不可变特性延伸到了宿主机操作系统层面。
  * **其他运维辅助工具**：
    * 涵盖跨平台网络构建（如 Weave、Flannel）和集中化日志路由分发工具（如 Heka）。
* **预留拓展补充空间**
  * > *[此处可补充：在众多编排工具的"战争"中，Kubernetes（K8s）最终确立绝对统治地位的技术和生态原因分析]*
